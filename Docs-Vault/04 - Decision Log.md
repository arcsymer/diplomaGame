## ADR-001 — Версия Unity: 6000.4.9f1 (изначально рекомендована 6.3 LTS)
**Дата:** 2026-06-10 · **Статус:** принято, дополнено в тот же день

**Проблема:** выбрать версию Unity для проекта со сроком сдачи (диплом).

**Варианты:**
1. Unity 6.0 LTS — поддержка до октября 2026 (кончится до защиты).
2. **Unity 6.3 LTS (6000.3.17f1)** — текущая LTS, поддержка до декабря 2027.
3. Unity 6.4 (Supported) — новые фичи (встроенный ECS, DirectStorage), но поддержка умрёт с выходом 6.5.

**Рекомендация ресёрча:** Unity 6.3 LTS — официальная рекомендация Unity для новых проектов; двухлетний хвост багфиксов покрывает весь срок диплома.

> **UPD 2026-06-10 — итоговое решение:** пользователь уже создал проект на **Unity 6000.4.9f1 (6.4, Supported)**. Остаёмся на 6.4: проект инициализирован, пакеты совместимы (URP 17.4, Cinemachine 3.1.6, Input System 1.19, AI Navigation 2.0.12), EditMode-тесты 3/3 зелёные. **Принятый риск:** поддержка 6.4 закончится с выходом 6.5 (~конец 2026); при блокирующих багах возможен переход на 6.3 LTS. Версию редактора не меняем до конца диплома.

Источники: [support schedule](https://unity.com/releases/unity-6/support) · [6.3 LTS blog](https://unity.com/blog/unity-6-3-lts-is-now-available)

## ADR-002 — Cinemachine 3.1, не 2.x
**Дата:** 2026-06-10 · **Статус:** принято (Фаза 0)

**Проблема:** Cinemachine 3.x кардинально сменил API относительно 2.x (большинство туториалов в сети — про 2.x).

**Выбор:** Cinemachine **3.1.x** — verified-версия для Unity 6.3. Работать только по документации 3.x: namespace `Unity.Cinemachine`, единый `CinemachineCamera` вместо `CinemachineVirtualCamera`/`FreeLook`, pipeline-компоненты как обычные MonoBehaviour, Channels вместо слоёв. **Причина:** 2.x — legacy; начинать новый проект на устаревшем API бессмысленно.

Источник: [Upgrade guide 2.x→3.x](https://docs.unity3d.com/Packages/com.unity.cinemachine@3.1/manual/CinemachineUpgradeFrom2.html)

## ADR-003 — UI: uGUI (Canvas + TMP) для HUD, UI Toolkit не используем в рантайме
**Дата:** 2026-06-10 · **Статус:** принято (Фаза 0)

**Проблема:** uGUI vs UI Toolkit для игрового UI (RTS HUD: миникарта, панель выделения, health bars над юнитами; TPS HUD: прицел, кулдауны; меню/настройки).

**Варианты:**
1. **uGUI всё** — зрелый, огромная база рецептов (миникарта = RawImage+RenderTexture, health bars = world-space canvas), твинится PrimeTween, дружит со сторонними ассетами.
2. UI Toolkit всё — современный стек, но world-space UI появился только в 6.2/6.3 (свежая фича = риск).
3. Гибрид (uGUI HUD + UI Toolkit меню) — две системы = двойная сложность стилей/анимаций.

**Выбор:** uGUI для всего. **Причина:** минимизация рисков в дедлайн-проекте; TextMeshPro теперь встроен в uGUI 2.0; официальная позиция Unity допускает uGUI как полностью поддерживаемый путь. Пересмотрим, только если упрёмся в производительность data-heavy экранов (не ожидается).

Источники: [UI comparison](https://docs.unity3d.com/Manual/UI-system-compare.html) · [World-space UI Toolkit](https://docs.unity3d.com/6000.3/Documentation/Manual/ui-systems/world-space-ui.html)

## ADR-004 — Твининг: PrimeTween вместо DOTween
**Дата:** 2026-06-10 · **Статус:** принято (Фаза 0)

**Варианты:** DOTween (популярнейший, но closed-source, аллокации, NRE при уничтожении целей) / **PrimeTween** (MIT, zero-allocation, безопасен при destroy) / LitMotion (MIT, быстрейший, data-oriented).

**Выбор:** PrimeTween. **Причина:** open-source MIT (чистая лицензионная история для диплома), zero-alloc (плюс к разделу о производительности), API проще LitMotion для UI-juice. DOTween отвергнут из-за closed-source и известных NRE-ловушек.

Источники: [PrimeTween](https://github.com/KyryloKuzyk/PrimeTween) · [Tween benchmark](https://github.com/AnnulusGames/TweenPerformance)

## ADR-006 — Аудио: только CC0/CC-BY библиотеки + локальная генерация; все облачные сервисы отвергнуты
**Дата:** 2026-06-10 (пересмотрено в тот же день под ADR-009) · **Статус:** принято (Фаза 0)

**Проблема:** выбрать пайплайн музыки, SFX и голосов юнитов с чистыми лицензиями при **нулевом бюджете** (ADR-009: без подписок, триалов, карт).

**Варианты и почему отпали:**
- **Udio** — скачивание треков отключено на всех тарифах после сделки с UMG → непригоден физически.
- **Suno** — нет официального API; free-тир без скачивания (сделка с Warner); платный Pro запрещён ADR-009.
- **ElevenLabs** (TTS/SFX/музыка) — freemium-подписка → запрещён ADR-009. *(Первая редакция этого ADR рекомендовала Starter $6/мес — отменено.)*
- **Stable Audio API / Azure TTS F0** — платный API / требует облачный аккаунт → запрещены ADR-009.

**Выбор (всё $0, без аккаунтов):**
- Музыка: incompetech/Kevin MacLeod (CC-BY 4.0, кредит в титрах) + Pixabay + OpenGameArt (CC0/CC-BY); опционально локальная генерация **Stable Audio Open** (открытые веса, Stability Community License — свободно при < $1M выручки).
- SFX: Kenney (CC0) + Sonniss GDC 2026 (royalty-free) ≈ 90%; точечно Freesound (бесплатный API, фильтр CC0/CC-BY); ретро/UI — sfxr/Bfxr/ChipTone локально.
- Голоса юнитов: путь №1 — **Kenney Voiceover Pack / (Fighter)** (CC0, готовые ack-реплики); путь №2 — **Piper TTS** локально (код GPL, лицензию конкретного голоса проверять на HF rhasspy/piper-voices).
- **MusicGen** — только запасной вариант: веса **CC-BY-NC** (для учебного диплома допустимо, для коммерческого распространения — нет; фиксирую согласно ADR-009/zero-budget §Лицензии).

**Причина:** соответствие нулевому бюджету; лицензионная чистота для публичного репозитория; Kenney Voiceover закрывает голоса вообще без генерации.

Источники: см. [[Research/00 - Prerequisites|Prerequisites, раздел B]]

## ADR-008 — VCS: выборочный LFS с первого коммита; CI — минимальный и опциональный
**Дата:** 2026-06-10 · **Статус:** принято (Фаза 0), развилка по CI за пользователем

**Выбор:**
- LFS включаем до первого коммита (миграция задним числом = переписывание истории), но **выборочно**: крупные бинарники в LFS, мелкие (<~5 МБ png/ogg) — обычный git. Причина: free-тир 10 GiB хранилища + 10 GiB трафика/мес; data packs отменены; каждый clone/CI-checkout жжёт трафик.
- Билд распространяем через **GitHub Releases** (2 GiB/файл, трафик бесплатен), не через LFS.
- CI (GameCI unity-builder@v4): Actions для публичных репо бесплатны, но нужен секрет UNITY_LICENSE (действие пользователя) и сборки медленные. ~~Рекомендация: стартуем без CI~~ → **UPD 2026-06-10: пользователь решил делать CI/CD.** Внедрено: `tests.yml` (EditMode+PlayMode на push/PR) + `build-release.yml` (Windows x64 по тегу `v*` → автоматический GitHub Release). Основной путь сборки всё равно локальный (Project Forge); CI — верификация и релизы. До добавления секретов `UNITY_LICENSE`/`UNITY_EMAIL`/`UNITY_PASSWORD` workflow будет падать — ожидаемо. *(UPD: секреты добавлены 2026-06-10.)*

> **UPD 2026-06-10 (CI-образ):** Docker-образа `unityci/editor` для 6000.4.9f1 не существует (GameCI пропустил этот патч: есть 6000.4.8f1 и 6000.4.10f1). Решение: `customImage` с пином на **6000.4.10f1** в обоих workflow. Принятый риск: расхождение локального редактора (.9) и CI (.10) в пределах патч-версии — несущественно; при удобном случае локальный редактор можно обновить до 6000.4.10f1 для полного паритета.

Источники: см. [[Research/00 - Prerequisites|Prerequisites, раздел E]]

## ADR-009 — Жёсткое ограничение: нулевой бюджет проекта
**Дата:** 2026-06-10 · **Статус:** принято (директива пользователя, `zero-budget-constraint.md`) · **Приоритет: выше всех остальных решений**

**Суть:** бюджет проекта — $0. Запрещены: покупки, подписки, платные API, платные ассеты, а также **любые триалы/freemium с картой, сроком или конвертацией в платное**. Разрешены только: FOSS (бесплатно навсегда, без карты), локальная/self-hosted генерация, CC0/свободные ассеты, постоянно-бесплатные сервисы без карты (Unity Personal, GitHub, Git LFS free tier, gh CLI, Obsidian, Node.js).

**Последствия для ранее принятых решений:**
- ADR-006 пересмотрен: ElevenLabs/Stable Audio API/Azure исключены → CC0/CC-BY библиотеки + локальная генерация (Stable Audio Open, Piper).
- ADR-007 пересмотрен: Synty исключён → строго CC0-арт.
- Лицензионная политика: приоритет CC0 > CC-BY (с атрибуцией) > CC-NC (только если нет свободного аналога, помечать отдельно — для учебной работы допустимо, для коммерческого распространения нет). Всё внешнее — в `Licenses & Attribution.md`; ничего с неясной лицензией.

**Проверка перед выбором любого инструмента:** бесплатен навсегда? без карты? не триал? Если бесплатен только trial — отклонить и найти FOSS/CC0-аналог с фиксацией здесь.

## ADR-010 — Цели боевого ИИ: `Health` как общая сущность юнитов и зданий
**Дата:** 2026-06-11 (M10) · **Статус:** принято

**Проблема:** В исходной реализации `UnitCombat.ScanForTarget()` искал цели только через `UnitRegistry`. Здания в него не входили. Следствие: юниты не атаковали здания противника — волна доходила до базы и стояла. Победа волнами была принципиально недостижима (M10-BUG-001, выявлен интеграционными тестами `MatchSimulationTests`).

**Варианты:**
1. Добавить здания в `UnitRegistry` — смешение ответственности.
2. Дать `UnitCombat` отдельный список зданий — дублирование логики поиска ближайшего.
3. **Унифицировать через `Health`**: `UnitCombat` собирает кандидатов из `UnitRegistry` и `BuildingRegistry`, оборачивает их в общий буфер `List<Health>` + `List<Vector3>`, применяет единый `CombatLogic.FindNearestTargetIndex`.

**Выбор:** вариант 3. **Причина:** `Health` — уже общий компонент юнитов и зданий (оба реализуют `IDamageable`). Рефакторинг не меняет интерфейс — меняется только источник списка кандидатов. Дополнительно: для зданий добавлен буфер `BuildingAttackBuffer = 2f` к `attackRange` (здания крупные, без буфера юнит не «входит» в dистанцию атаки). Определение «цель — здание» — по отсутствию `NavMeshAgent` на цели (дёшево, без тега).

## ADR-011 — Авто-скриншоты: `ScreenshotDirector` + CLI-аргумент `-autoshot`
**Дата:** 2026-06-11 (M11) · **Статус:** принято

**Проблема:** скриншоты для README нужны из обоих режимов (RTS и TPS). Делать их вручную — ненадёжно (разные запуски, разные состояния). CI-сборка должна уметь произвести скриншоты без интерактивности.

**Варианты:**
1. Ручные скриншоты в редакторе (ненадёжно, не воспроизводимо).
2. Unity Test Framework с `ScreenCapture` (ограничен PlayMode, нет контроля над Cinemachine-блендом).
3. **`ScreenshotDirector` — компонент на сцене, активируется только при наличии CLI-аргумента `-autoshot <dir>`**.

**Выбор:** вариант 3. Реализация: компонент засыпает 2.5 с (прогрев сцены) → снимает RTS → `SetMode(Tps)` → ждёт 3.5 с (кроссфейд Cinemachine ~2 с + запас) → снимает TPS → `Application.Quit()`. `[RuntimeInitializeOnLoadMethod]` обеспечивает переход MainMenu → Sandbox при старте с `-autoshot`. **Причина:** воспроизводимо, не мешает нормальной работе, встроено в релизный билд без условной компиляции.

## ADR-012 — Нормализация масштабов FBX по баундам вместо ручной подгонки
**Дата:** 2026-06-11 (M11) · **Статус:** принято

**Проблема:** Kenney FBX-модели имеют разные единицы масштаба (часть в сантиметрах, часть в метрах, часть в других единицах). Ручная подгонка `localScale` каждого префаба — неидемпотентная работа, которую нельзя воспроизвести batch-вызовом.

**Варианты:**
1. Ручной подбор `Scale Factor` в Import Settings для каждой модели.
2. Нормализация через `ModelImporter.globalScale` в пакетном Editor-скрипте.
3. **Нормализация по баундам в Forge**: при Setup Visual читаем `Renderer.bounds.size` нового Visual-child и вычисляем `localScale` так, чтобы наибольшее из измерений bounding box соответствовало заданному целевому размеру (1 м для юнитов, 2 м для зданий).

**Выбор:** вариант 3. **Причина:** идемпотентен (Project Forge кнопка), не трогает Import Settings (не сломает GUID), автоматически корректирует любой будущий FBX. Единственный минус — bounding box может быть не идеальным (Т-поза персонажа шире вертикали), поэтому для юнитов берём наибольшее измерение и ограничиваем снизу по оси Y. На практике погрешность < 10% — приемлемо для диплома.

## ADR-013 — Реальные способности героя 2–4: Ударная волна, Ремонтное поле, Перегрузка
**Дата:** 2026-06-11 (Сессия 02, фаза улучшения) · **Статус:** принято

**Проблема:** DoD промпта v2 требует «2–4 способности» героя; фактически в v1.0.0 реален только Dash, слоты 2–4 — `Placeholder` с `Debug.Log`. Гэп-анализ (Improvements/01) пометил это как Р-1 — главный разрыв DoD.

**Варианты дизайна слотов (с учётом темы диплома — сниженный микроменеджмент):**
1. Классические «стрелковые» абилки (граната по точке прицела, мина, турель) — повышают требования к прицеливанию и APM.
2. **Ауры/импульсы вокруг героя**: AoE-урон, массовое лечение, самобафф — эффект применяется одной кнопкой без прицеливания.
3. Командные абилки RTS-уровня (глобальный приказ, сканер) — дублируют существующие армейские приказы.

**Выбор:** вариант 2 — три способности «вокруг героя»:
- **Слот 2 «Ударная волна»** (`Shockwave`): 40 урона всем врагам в радиусе 6 м, кулдаун 10 с. Герой влияет на массовый бой одной кнопкой.
- **Слот 3 «Ремонтное поле»** (`RepairField`): лечение 30 HP союзным юнитам в радиусе 8 м, кулдаун 12 с. **Ключевое анти-микро-решение:** вместо ручного отвода раненых юнитов (микро в SC2) игрок лечит армию на месте — позиция героя становится тактическим решением вместо APM.
- **Слот 4 «Перегрузка»** (`Overcharge`): ×2 скорострельность и ×1.5 урон героя на 5 с, кулдаун 15 с. Усиливает TPS-петлю без новых входов.

**Причина:** все три применяются одной кнопкой без прицеливания → нулевой прирост APM при росте тактической глубины (когда войти в толпу, когда лечить, когда бафф). Баланс — в `ManagersTab.EnsureAbilityAsset` (Forge — источник истины), данные — `AbilityData` (data-driven, без хардкода). Эффекты переиспользуют существующие системы: `UnitRegistry` (выбор целей), `Health.TakeDamage`/новый `Health.Heal` (клامп — `CombatLogic.ClampHeal`), `HeroShooter.ApplyOvercharge`. VFX — переиспользование пулов `VfxManager` по событию `AbilityCast` (без новых ассетов — нулевой бюджет).

**Верификация:** EditMode — `AbilityEffectLogicTests` (радиус, клампы, кулдаун-математика); PlayMode — `AbilityTests` (урон/лечение/бафф по фракциям и радиусам).

## ADR-014 — Измерительный харнесс фазы улучшения: PerfProbe + Balance-плейтесты + вкладка Improve
**Дата:** 2026-06-11 (Сессия 02, фаза улучшения) · **Статус:** принято

**Проблема:** промпт v3 (§3, §8.2) требует baseline-метрики (FPS/frame time, GC, винрейты AI vs AI) и харнессы до первого улучшения. В v1.0.0 измерительной инфраструктуры нет (Р-2/Р-3 гэп-анализа).

**Варианты:**
1. Unity Profiler вручную в редакторе — не воспроизводимо, не batch, противоречит правилу Forge.
2. Сторонние перф-пакеты (Unity Performance Testing Extension) — тянет зависимость, формат заточен под CI-датабазы Unity.
3. **Собственный лёгкий харнесс**: `PerfStatsLogic` (чистая статика — avg/p95/worst/FPS) + `PerfProbe` (MonoBehaviour, кольцевой буфер `unscaledDeltaTime`, GC-дельта) + PlayMode-тесты категории `Balance` (армия против армии под боевой FSM) с экспортом JSON в `Docs-Vault/Stats/` + вкладка **Improve** в Forge (`ForgeBatch.ExportImprovementMetrics`) — накопительная таблица `Improvements/Metrics.md`.

**Выбор:** вариант 3. **Причина:** нулевые зависимости, полная воспроизводимость одной CLI-строкой (`-runTests -testCategory Balance`), чистая логика тестируется в EditMode, история метрик накапливается в Obsidian (требование v3 §6 «метрики не переписываются»). Сценарии: зеркальный бой 8v8 (проверка симметрии), асимметричный 8v8 Marine vs Grunt (фактический баланс), стресс 20v20 с перф-записью (baseline под нагрузкой).
